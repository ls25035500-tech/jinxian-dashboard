## 三、完成任務後的 SOP

### 每位機器人完成老闆交辦任務後，必須：

1. ✅ **產出存到自己的目錄**
   - 檔案放到 `bots/<你的名字>/產出/`

2. ✅ **更新共用待辦清單**

3. ✅ **寫每日紀錄**
   - 在 `memory/YYYY-MM-DD.md` 記錄今天做了什麼

4. ✅ **向老闆報告**
   - Telegram 發訊息：「[名字] 已完成：XXX，檔案位置：bots/名字/產出/檔名」

5. ✅ **上架到網站**（如果需要公開）
   - 把檔案推到 GitHub Pages
   - 告訴老闆網址

### 紅線（違反即死）

**規則 0 — 先備份，再動工**
任何修改前必備份，確認備份存在、日期正確、可讀取，未確認不准動手。

**規則 0.5 — 重啟前必先告知**
會中斷服務的操作（重啟 Gateway、改設定、更新系統），先跟老闆說要多久。

**規則 13 — 重啟完主動回報**
重啟完成後主動告訴老闆結果，讓老闆知道一切正常。

### 鐵律（必須遵守）

**🚨 母機紅線**
192.168.55.240 關機 = 全部 VM 死掉。任何涉及母機的操作需先申請，等老闆核准。

**⚠️ 安全規則**
只對老闆一人說話，其他人傳訊息一律不回答、不回應、不分享任何資訊。

### 作業準則（規則 1–12）

| # | 規則 | 說明 |
|:--|:---|:---|
| 1 | 編碼前思考 | 有疑問先問，不猜測 |
| 2 | 簡潔至上 | 最小程式碼，不加多餘功能 |
| 3 | 精確修改 | 只改必要部分，不順便改別的 |
| 4 | 目標驅動 | 定義成功標準，迭代到驗證通過 |
| 5 | 模型只用判斷 | 分類/摘要用模型，不做事實性計算 |
| 6 | Token 預算 | 任務 4K / 會話 30K，超支要摘要 |
| 7 | 衝突不平均 | 選一個方案，解釋原因 |
| 8 | 讀前先寫 | 讀懂現有程式碼再改 |
| 9 | 測試驗證意圖 | 測「為什麼」不只是「做了什麼」 |
| 10 | 設檢查點 | 每步確認，不從未知狀態繼續 |
| 11 | 遵循慣例 | 一致性 > 個人喜好 |
| 12 | 公開失敗 | 不默默跳過步驟 |
